# 如何用 AI 学习开源项目

---

## 一、核心思路

传统学开源项目：啃源码 → 看不懂 → 放弃。  
AI 辅助学法：**AI 当导航 + 翻译 + 导师**，帮你快速穿透项目。

> 把 AI 想象成一个**对这个项目了如指掌的同事**，你随时可以问。

---

## 二、五步学习法

### 第 1 步：让 AI 帮你建立全局地图

拿到一个陌生项目，先用 AI 快速摸清整体：

```
帮我分析这个项目：
- 这是什么项目？解决什么问题？
- 整体架构是什么？用了哪些核心技术和设计模式？
- 主要模块有哪些？各模块职责是什么？
- 项目的入口在哪里？调用链路是怎样的？
```

如果你已经 clone 到本地，打开关键文件（`README.md`、`Program.cs`、核心接口），AI 能看到更多上下文。

> **这一步的目标**：不看一行代码细节，先知道"这个项目是什么、怎么组织的"。

---

### 第 2 步：从入口开始，逐层往下挖

```
从 Program.cs / Startup.cs 开始，帮我逐行解释：
- 每一段代码在干什么
- 注册了哪些服务，为什么这么注册
- 中间件管道的执行顺序及其原因
```

然后继续：

```
接下来帮我分析 [UserController] 的完整请求处理流程：
从 HTTP 请求进来 → Controller → Service → Repository → 数据库，
每一步做了什么，数据是怎么流转的。
```

> **这一步的目标**：理解一条完整链路，建立"数据怎么流"的直觉。

---

### 第 3 步：挑一个你感兴趣的功能深挖

```
我想深入理解 [JWT 鉴权] 在这个项目中是怎么实现的。
帮我：
1. 找出所有相关的文件和类
2. 画出认证流程（从登录到 Token 验证）
3. 解释每一步的设计考量（为什么这么写，不那样写）
```

---

### 第 4 步：动手改造，加深理解

```
我想在这个项目基础上加一个 [操作日志记录] 功能。
帮我分析：
1. 应该加在哪个层？为什么？
2. 需要改哪些文件？
3. 给出现有风格一致的代码实现
4. 解释每个改动的原因
```

> **这一步的目标**：通过"改"来验证你是不是真懂了。能改对 = 学透了。

---

### 第 5 步：横向对比，举一反三

```
这个项目的 [仓储模式] 实现和 ABP 框架的仓储模式有什么不同？
优缺点分别是什么？
```

或者：

```
我同时打开了两个开源项目（A 和 B），
帮我对比它们的 [依赖注入 / 异常处理 / 项目分层] 方式的异同。
```

## 三、针对不同类型开源项目的提问模板

### Web API / 后端项目（如 eShopOnContainers、ABP）

```
帮我分析这个 Web API 项目的架构：
- 分层方式（Clean Architecture / 传统三层 / DDD？）
- 依赖注入的使用方式
- 中间件和 Filter 的使用
- 异常处理和日志策略
- 数据库访问层的实现（EF Core / Dapper / 原生 SQL？）
```

### 前端项目（如 Ant Design Pro、Vue Element Admin）

```
帮我分析这个前端项目：
- 路由和权限控制是怎么设计的？
- 状态管理用的是什么方案？为什么？
- 组件封装的模式和复用策略
- API 请求封装和拦截器怎么写的？
```

### 工具/类库（如 AutoMapper、Polly、MediatR）

```
帮我分析 [AutoMapper] 的核心设计：
- 核心接口和类有哪些？
- 它是怎么实现对象映射的？
- 用到了哪些设计模式？
- 如果让我自己写一个简化版，最小可行实现是怎样的？
```

## 四、实战例子：用 VS Code + AI 学 ABP 框架

### 步骤演示

```markdown
# 第一轮：先摸全局
"我刚 clone 了 ABP 框架的源码，帮我从解决方案结构分析它的模块划分和组织方式。"

# 第二轮：看启动流程
"打开 HttpApi.Host 项目的 Program.cs，帮我分析启动时做了什么。"

# 第三轮：深入模块系统
"ABP 的模块系统（AbpModule）是怎么实现的？帮我找出核心类并解释加载机制。"

# 第四轮：看具体功能
"帮我分析 ABP 的审计日志是怎么实现的，从拦截器到写入数据库的完整链路。"

# 第五轮：动手实践
"参考 ABP 的模块化设计思路，帮我在我的项目里实现一个简化版的插件加载系统。"

# 第六轮：对比学习
"ABP 的依赖注入封装和原生 .NET DI 有什么增强？为什么这么设计？"
```

## 五、高效技巧

### 1. 用 Agent 模式自动探索项目

```
帮我找出项目里所有使用了 [策略模式] 的地方，
列出类名、文件路径、以及各自负责什么策略。
```

→ Agent 模式会自己搜文件、读代码、整理给你。

### 2. 画流程图/类图

```
帮我分析 [订单创建流程]，画出关键类和方法的调用关系。

用 Mermaid 格式输出：
- 时序图：展示方法调用顺序
- 类图：展示核心类的关系
```

AI 可以直接在 VS Code 中渲染 Mermaid 图。

### 3. 不懂就问，别跳过

```
这行 var result = await _mediator.Send(command); 
里面的 MediatR 是什么？把"命令"发给谁了？
```

### 4. 让它用你的水平解释

```
我刚学 C# 一年，用我能看懂的语言解释这段代码。
```

或者：

```
用中级开发者的水平解释，别跳步骤。
```

### 5. 对比源码与你自己的写法

```
这段开源项目的实现方式和我之前写的有什么不同？

开源项目写法：
```csharp
// 粘贴开源代码
```

我的写法：
```csharp
// 粘贴你的代码
```

帮我分析差异和各自的优缺点。
```

## 六、新手常见误区

| ❌ 错误做法 | ✅ 正确做法 |
|------|------|
| 从头到尾一行行看 | 先看架构、入口、核心链路，再挑兴趣点深挖 |
| 看不懂硬看 | 随时打断，问 AI "这段在干什么" |
| 只看不改 | 动手加功能/改逻辑，才能真正理解 |
| 一次想搞懂整个项目 | 一次一个模块，学透再下一个 |
| 只看代码不画图 | 让 AI 画出调用链，可视化理解更快 |
| 不问"为什么这么设计" | 设计决策比代码本身更有价值 |

## 七、推荐学习路径

```
第 1 天：AI 分析项目全貌 → 画出架构图 → 理解模块划分
第 2 天：跟踪一条完整请求链路（从入口到数据库）
第 3 天：深入一个你感兴趣的功能模块
第 4 天：模仿写一个简化版 → 对比差距
第 5 天：修改/扩展一个功能 → 提交 PR 给原项目
```

## 九、你的第一句实操提示词

挑一个开源项目 clone 下来，用 VS Code 打开，然后说：

> 我刚打开 [项目名] 的源码，目录在 [路径]。  
> 先帮我分析这个项目的：整体架构、核心模块、入口文件、技术栈。  
> 然后帮我选一条最容易上手的调用链路，带我一步步走一遍。
