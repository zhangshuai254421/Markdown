# AI 辅助 C# 开发学习文档

---

## 一、Copilot 四种模式区别

| 模式 | 核心特点 | 是否修改文件 | 是否自主探索 |
|------|---------|:----------:|:----------:|
| **Ask** | 问答式，一次性回答 | ❌ 只读 | ❌ |
| **Edit** | 直接编辑指定文件 | ✅ | ❌ |
| **Agent** | 自主多步执行 | ✅ | ✅ |
| **Plan** | 先出计划，确认后再执行 | ✅（确认后） | ✅ |

### Ask（问答模式）
- 你问 → AI 答，纯对话，只读不碰文件
- 适合：理解代码、问 API 用法、代码审查、学习类问题

### Edit（编辑模式）
- 你指定改哪里 → AI 直接改，目标明确、改动范围小
- 适合：修 bug、重构某个函数、加小功能、改名格式化

### Agent（代理模式）
- 你给目标 → AI 自主探索代码库、规划步骤、逐步执行
- 适合：跨文件复杂重构、新增完整模块、需先探索项目的任务

### Plan（计划模式）
- 相当于 Agent + 人工审核关，先出计划你确认后再执行
- 适合：高风险改动、不确定最佳方案、需先对齐思路

> **口诀**：只问不改→Ask | 小改一下→Edit | 大活全包→Agent | 大活但想先看看方案→Plan

## 二、C# 开发提示词写作指南

### 通用公式

> **角色 + 上下文 + 任务 + 约束**

```markdown
## 角色（可选）
你是一个资深 C# 后端开发工程师

## 上下文
我正在开发一个 .NET 8 Web API 项目，使用 Entity Framework Core 操作 SQL Server。

## 任务
帮我创建一个 UserController，包含注册、登录、获取用户信息的接口。

## 约束
- 使用 RESTful 风格
- 密码需要 BCrypt 加密
- 返回统一的 ApiResponse<T> 格式
- 加上参数校验
```

## 三、C# 常见场景模板

### 1. 新建功能

```

项目使用 [技术栈: .NET 8, EF Core, Dapper 等]。
要求：
- [约束1]
- [约束2]
```

### 2. 代码重构

```
帮我重构下面的代码，目标是 [提高可读性 / 提取接口 / 用设计模式改造]。
当前问题：[重复代码太多 / 耦合度高 / 方法太长]

```csharp
// 粘贴你的代码
```
```

### 3. 代码审查

```
帮我审查这段 C# 代码，关注 [性能 / 安全性 / 空引用 / 异常处理 / 线程安全]。
并给出改进建议和修改后的代码。
```

### 4. Bug 排查

```
这段代码预期 [期望行为]，但实际 [实际行为]。
帮我找出问题并修复：

```csharp
// 有问题的代码
```
```

### 5. 写单元测试

```
为下面的方法写 xUnit 单元测试，覆盖 [正常路径 / 边界值 / 异常情况]。
使用 Moq 模拟依赖：

```csharp
// 要测试的方法
```
```

### 6. 数据库相关（EF Core）

```

实体包括：Order、OrderItem、Product。
关系：[一对多 / 多对多 等]
要求使用 Fluent API 配置。
```

### 7. 解释代码

```
帮我解释这段 C# 代码的 [设计思路 / 执行流程 / Linq 逻辑]：

```csharp
// 代码
```
```

## 四、实战对比

### ❌ 不好的提示词

> "帮我写个登录"

模糊、无上下文、无约束。

### ✅ 好的提示词

> 在 .NET 8 Web API 项目中，帮我写一个 JWT 登录接口。
> - 接收 username + password
> - 调用已有的 `IUserService.AuthenticateAsync` 验证
> - 返回 `{ token, expiresAt }`
> - 加上输入校验和统一异常处理

## 五、新手最佳实践

| 技巧 | 说明 |
|------|------|
| **先说项目背景** | ".NET 8 + EF Core + PostgreSQL" 一句话交代技术栈 |
| **给约束，别给废话** | "用异步方法"、"遵循项目现有风格"、"别用第三方库" |
| **善用追问** | 回答不满意→ "还能再简洁点吗" / "解释一下为什么要用 `await using`" |
| **把上下文带上** | 打开相关文件在编辑器里，AI 能看到它们 |
| **复杂任务用 Plan 模式** | 先看方案再执行，避免返工 |

## 七、左下角智能体选择器说明

### 各选项含义

| 选项 | 含义 | 适用场景 |
|------|------|----------|
| **Copilot** | GitHub 云端默认模型 | 日常开发，最推荐，速度快效果好 |
| **本地** | 本机运行的小模型 | 离线环境、敏感代码不想上传云端时用 |
| **CLI** | 命令行终端模式 | 在终端里用 AI，不走聊天面板 |
| **云** | 其他云端模型（如 GPT-4o 等） | 需要更强推理能力、处理复杂逻辑时切换 |

> **新手建议**：90% 的情况用默认的 `Copilot` 就行，不用纠结切换。  
> 公司代码不能传外网 → 切 **本地** | Copilot 回答不够好 → 切 **云** | 想在终端里直接问 AI → 用 **CLI**

## 八、左下角 vs 右边模型选择器的区别

```
左下角（智能体）               右边聊天框（模型）
┌──────────────┐               ┌──────────────────┐
│ Copilot      │               │ DeepSeek V4 Pro  │
│ 本地          │               │ GPT-4o           │
│ CLI          │               │ Claude 3.5       │
│ 云           │               │ ...              │
└──────────────┘               └──────────────────┘
   "谁来跑" →                    "跑哪个大脑" →
   决定运行位置                  决定具体模型
```

### 形象类比

| 层级 | 类比 | 决定什么 |
|------|------|----------|
| **左下角（智能体）** | 选外卖平台（美团/饿了么/堂食） | AI 在**哪里运行** |
| **右边（模型）** | 选具体餐厅（肯德基/麦当劳/海底捞） | 用**哪个具体模型** |

例如：左边选 **Copilot** + 右边选 **DeepSeek** = 通过 GitHub Copilot 平台，调用 DeepSeek 模型来回答你。

> **新手建议**：右边模型一般不用动，DeepSeek 挺好用的。回答质量不满意再切右边试试其他模型即可。
